Task 执行、事件与恢复模型

磁盘日志是恢复事实源，Session completion event 驱动运行时；界面不自行推测终态。

标准执行序列
TaskEditor 验证→ 保存 task.yaml→ 创建/采用父 Session→ TaskRunner.run ↓ 调度 ready nodes→ 创建 Child Sessions→ onSessionComplete→ 写 output + run-log ↓ 全部节点完成→ Orchestrator verdict→ PASS / bounded repair→ 完成 + TW 同步
运行状态机

running

按依赖与 max_parallel 调度。

paused

不再派发新节点；已运行节点继续。

verifying

节点完成，等待父 Session 判定。

repairing

FAIL 后仅重跑 repair frontier，最多 10 次。

completed

PASS，输出与同步完成。

failed/stopped

明确终态，不自动伪装成功。

失败与一致性策略

本地事务边界

  • 先验证，再原子写 task.yaml / project config
  • Session 字段更新成功后才广播 renderer
  • 拖拽采用 optimistic UI；IPC 失败立即回滚
  • run-log 先落盘，再更新内存快照与 UI

外部同步边界

  • Teambition 不参与本地原子事务，本地完成不因远端超时回滚
  • 同步失败记录 pending/conflict/stale，可人工重试
  • 写操作做 preflight capability 与绑定校验
  • 同一 binding 使用幂等键，避免重复领取和重复评论
资源护栏

并发同时受 TaskSpec.max_parallel 与应用全局上限约束；token_budget 达限时自动 pause；重试、repair、verdict 格式重问均有独立上限；stop 会取消活动 Child Session 并写入明确日志。
