## 1. 问题

### 1.1 手机端远程编程（类 Cursor Remote / 移动续跑）

1. 现网是 **Electron 桌面 + 本地优先**（`~/.luxagents[-dev]/`）；离开工位无法看编排 live、批权限、续跑会话。  
2. 「手机当 thin client」需要：鉴权、会话/流式桥、权限/AskUser 移动 UI、与桌面单写者冲突策略——现网 IPC 模型未覆盖。  
3. 若只做只读看板/通知，价值有限；若做完整编程，范围会膨胀到远程 cwd / 终端 / 文件编辑。

### 1.2 Repo Wiki（类阿里 Qoder / 同类「仓库知识层」）

1. 项目中心已有项目壳与看板，但缺少 **仓库级、可检索、可随代码演进的知识层**（结构、模块说明、决策记录、变更摘要）。  
2. Agent / 专家 preamble 目前靠 IDENTITY/SOUL/RULES + skill slug；缺「读仓库 wiki」的一等入口时，长项目上下文仍靠对话堆。  
3. Wiki 若做成在线 DB 或强耦合云，会与「本地文件优先、可移植」冲突。

## 3. 边界（In / Out）

### 3.1 手机远程编程

**可能纳入（仅作问题边界，未选型）**

- 移动端查看：Agent/Kanban 会话列表、编排条进度、流式摘要  
- 移动端交互：权限批准/拒绝、AskUser、停止/续跑  
- 桌面（或本机 Agent 宿主）保持执行面；手机不跑 SDK binary  

**明确不做（本占位生命周期内）**

- 不在本文件拍板：推送通道（APNs/FCM）、自建 relay、还是 SSH/隧道  
- 不做完整移动 IDE（多文件编辑器、本地终端模拟器一等公民）  
- 不做「无桌面宿主」的纯云端 Agent 执行（与本地优先冲突，需另立产品决策）  
- 不改现有 Electron IPC 四件套为默认移动协议（须另定 API / auth）

### 3.2 Repo Wiki

**可能纳入（仅作问题边界，未选型）**

- 项目下 `wiki/`（或等价）Markdown / 索引；项目中心入口  
- Agent / 专家可引用 wiki slug 或路径（类 skill 引用，非整库内联）  
- 可选：由任务/会话触发的「更新 wiki 某页」工作流（仍文件落盘）  

**明确不做（本占位生命周期内）**

- 不上独立 Wiki 数据库或强制 SaaS  
- 不做与 Git hosting（GitHub Wiki / GitLab）的双向同步 MVP  
- 不做自动全仓扫描生成「永不过时」知识图谱的承诺（生成策略须另 design）  
- 不与蜂群 / Bot 绑定为前置条件

## 6. 开放问题（开 brainstorm 时再问）

1. 手机端 MVP：只读监控 vs 权限+续跑 vs 轻量发消息？

5. 「类 Qoder」要对齐的是产品概念（仓库知识层）还是具体交互（需对照竞品再锁）？
