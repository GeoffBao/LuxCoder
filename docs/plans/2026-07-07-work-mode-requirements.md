## 1. 背景与定位

### 1.1 背景

公司五化目标（系统化 / 标准化 / 货架化 / 工具化 / 客观化）要求以 LuxAgents 为唯一产品载体落地。行业侧，腾讯 WorkBuddy（2026-03，CodeBuddy 同架构外扩）验证了"编程 agent 内核 + 工作 agent 外壳"双模同构路线；差异在于它做横向通用办公，我们做**纵向研发闭环**（TW → 文档 → 审核 → 编码 → Gerrit → 回写），不与其在通用办公能力上对标。

本文档合并此前两条设计线：

| 设计线 | 核心贡献 | 保留内容 |
|---|---|---|
| SDD 看板线（v1.2，已入库） | 贴合 TB 审核实操的九阶段状态机、产出物目录约定、work-sdd Skill、手动兜底流程 | 全部保留为状态模型与 MVP 交互基础 |
| 闭环编排线（LuxBuddy v0.9） | 自动化闭环引擎、THO 契约、遥测第一、审批 gate、KPI 口径 | 全部保留为自动化层与度量体系 |

### 1.2 产品定位

**一个产品，三个模式**：Chat（对话）/ Code（编程 Agent）/ **Work（任务驱动的研发闭环编排）**。

核心边界原则：**Work 模式调用 Code 模式，但不管控它**。两者之间只有一份契约——Task Handoff Object（THO）。

Work 模式**不是另一个聊天框**，第一公民是"任务"：
- 主界面 = 任务看板（数据源为本地状态库，与 TW 双向同步）
- 点开任务 = 运行视图（阶段进度、产出物文档、Code 子会话实时日志、遥测）
- 对话框只作为任务内干预通道，不承担任务发起职能

### 1.3 任务来源（双通道）

- **TW 通道（主）**：TW 是团队任务的 SSOT，Work 模式通过 TW user-mcp 拉取与回写，不允许出现"只存在于 LuxAgents 里的团队任务"
- **本地通道（兜底）**：手动创建任务（`origin: 'local'`），服务于无 TB 环境 / 个人任务 / 演示场景；**本地任务不参与 KPI 遥测统计**

## 2. 核心设计哲学 — SDD（Spec-Driven Development）

先写规格再让 AI 写代码：`specify → plan → implement → verify`。人类在两个文档审核 gate 与代码 CR 处把关，AI 承担撰写与执行。

### 2.1 九阶段状态机（权威状态模型）

**需求 / Feature 类型（完整流程）：**

```
pending
  → req-writing        需求文档撰写（AI 生成）
    → req-reviewing    需求文档审核 gate ──驳回→ req-writing
  → design-writing     方案设计撰写（AI 生成）
    → design-reviewing 方案审核 gate ──驳回→ design-writing
  → coding             编码（THO 调用 Code 模式）
    → gerrit-reviewing Gerrit CR ──驳回→ coding（retries++）
  → pending-writeup    经验文档撰写，回写 TW
    → completed
```

**Bug / Task 类型（捷径流程）**：`pending → coding → gerrit-reviewing → pending-writeup → completed`

**终态**：`completed` / `cancelled`（进归档视图，不占看板）

### 2.2 自动化程度：手动可用，自动增强

每个阶段迁移都有两种触发方式，同一状态机：

| 触发方式 | 说明 |
|---|---|
| 手动 | 详情面板的流转按钮（v1.2 线全部保留）——第一天就可用，不依赖任何外部系统 |
| 自动 | 编排引擎驱动：Spec 自动生成后迁入审核态、审批通过自动进入下一撰写/编码、Code 会话完成自动迁移、TW 状态自动回写 |

M1 自动化覆盖：TW 拉取、req/design 文档生成、审批 gate、THO 编码、回写。Gerrit / 自动化测试节点 M2 接入前，对应迁移用手动按钮过渡。

## 3. 用户与角色

| 角色 | 使用方式 |
|---|---|
| SPM | 在 TW 建任务拆需求；在看板看任务全景 |
| SE / 专家 | 审批需求文档与方案文档（gate）；处理失败回环问题清单 |
| 开发（N≤3） | 监督任务执行；复杂任务切到 Code 模式手工介入 |
| 测试（N≤3） | 消费测试报告；维护自动化测试任务（M2） |
| 模块带头人 | 配置模块 Work Bot 模板：skills / 知识库 / 工具集（P2） |

## 4. 需求清单

### P0 — M1（闭环 demo：TW → 文档 → 审批 → 编码 → 回写）

| ID | 需求 | 验收标准 |
|---|---|---|
| P0-1 | TW 任务拉取（MCP） | 通过 TW user-mcp 程序化拉取指定项目任务列表/详情/状态，落本地状态库；MCP 工具列表枚举完成，读写覆盖面确认 |
| P0-2 | 任务看板 UI | "待我处理"聚合带 + 状态列 + 380px 详情面板；卡片实时活动指示；归档视图；新建任务 Dialog（本地通道） |
| P0-3 | 文档生成 | req-writing / design-writing 阶段由 AI 按 work-sdd 模板生成文档，归档到任务目录 `.luxagents-tasks/{taskId}/` |
| P0-4 | 人工审批 gate | 文档生成后自动迁入审核态；审批人批准/驳回（驳回必填意见，回撰写态）；审批人、耗时、意见入遥测 |
| P0-5 | THO 调用 Code 模式 | 审批通过后以 THO 拉起 Agent 会话（`origin:'work'` 打标），取回结构化结果 |
| P0-6 | 结果回写 TW | 任务状态、产出摘要回写；失败时问题描述回写；MCP 写能力缺失则标记待回写 + REST 兜底排期 |
| P0-7 | 遥测埋点（不可延期） | 每任务记录：ID/来源/各节点耗时/token/skills 引用/重试/终态。两个采集点：THO 回传、节点迁移 |
| P0-8 | 状态持久化与恢复 | 每次迁移写 checkpoint + 审计日志；重启后从 checkpoint 恢复，编码会话终态对账 |
| P0-9 | work-sdd 默认 Skill | 需求/方案/经验文档模板打包为 default skill，随工作区自动分发 |

> P0-7 特别说明：遥测是客观化 KPI 的唯一数据源，历史数据无法事后重建，必须与 M1 同期上线，没有商量余地。

### P1 — M2（闭环补全）

| ID | 需求 |
|---|---|
| P1-1 | Gerrit 节点：产出自动 push 评审，评审状态回流 |
| P1-2 | Jenkins 节点：构建触发与结果回流（webhook 优先，轮询兜底） |
| P1-3 | 自动化测试节点：冒烟→回归，报告结构化回流 |
| P1-4 | 失败回环：测试失败带日志与问题清单回编码节点重试，次数进遥测 |
| P1-5 | 五类文档补全：任务文档 / 代码实现文档 / 测试文档全链路"生成→检查→归档" |
| P1-6 | Skill 注入增强：按任务模块自动选择工作区与 skills，引用记录进遥测 |
| P1-7 | 授权边界强制：Code 子会话只能访问 THO 声明的仓库路径（M1 已在契约中记录） |
| P1-8 | Spec 生成升级：由 provider 直连升级为只读 Agent 会话（可读 repo 上下文） |

## 5. 非目标（明确不做）

1. 不做通用办公能力（Excel/PPT/海报）——那是 WorkBuddy 的战场
2. 不在客户端再造任务管理系统——TW 是团队任务 SSOT；本地通道只是兜底且不进 KPI
3. M1/M2 不做服务端多租户编排——单机 + 本地持久化验证闭环
4. MVP 全中文 UI，不引入 i18n 框架
5. 不引入本地数据库——JSON/JSONL 文件方案，重新评估阈值见方案文档

## 6. KPI 与度量口径（防 gaming 预定义）

| 指标 | 口径 | 反 gaming 说明 |
|---|---|---|
| 任务 AI 完成率 | 通过回归测试**并合入**的任务数 / AI 参与任务总数 | 不以代码行数计，防拆任务灌水 |
| DVA 节降率 | (基线人力工时 − 实际人力工时) / 基线，按任务类型分桶 | 基线 M1 前用历史 TW 数据冻结 |
| Skill 有效引用率 | 被加载且对产出有实际贡献的 skill 次数 / 加载总次数 | 防"注册数量"型指标 |
| 失败回环收敛度 | 平均重试次数、重试后通过率 | 衡量闭环质量而非速度 |
| 审批质量 | 驳回率、审批耗时（进遥测可回溯） | 防审批 gate 走过场 |

## 7. 里程碑

| 里程碑 | 范围 | 周期（估） |
|---|---|---|
| M0 | 本文档 + 方案文档 v1.0 冻结 + UI/UX 高保真原型评审通过 | 数日 |

## 8. 风险

| 风险 | 应对 |
|---|---|
| TW user-mcp 写能力（更新状态/评论）未确认 | M1 第一周枚举工具列表验证；缺失则回写走 REST 开放平台兜底（appAccessToken 流程另行申请） |
| LangGraph.js 团队不熟悉 | 只有 WorkEngine 一个模块接触；契约隔离；图级测试全覆盖 |
| 审批 gate 变走过场 | 驳回率/审批耗时进遥测，SE 审批质量可回溯 |
| KPI 口径事后被推翻 | M1 前冻结口径（§6），先冻结再采集 |
| Code 会话与 Work 引擎状态漂移 | 启动对账机制（核对 agentSessionId 实际终态）为 P0-8 验收项 |
